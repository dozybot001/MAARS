Refine & Write

Refine and Write share the same TeamStage base class and an IterationState-driven Multi-Agent loop. Symmetric at the loop level, they differ only in configuration. Write additionally overrides _execute() to append Polish and Metadata sub-steps.

01One shape, two roles

A primary agent drafts, a reviewer agent critiques, and an IterationState tracks the delta between them. Python runtime drives the loop. The LLM never decides when to stop — the runtime does, by inspecting whether issues has reached zero.

02IterationState

Three fields. Bounded context. No growth with iteration count.

@dataclass
class IterationState:
    draft: str              # latest full content (proposal / paper)
    issues: list[dict]      # [{section, problem, suggestion}]
    iteration: int          # current round
    _next_id: int = 1       # Issue ID counter (I1, I2, I3...)
Copied ✓

Update rules

  • draft: produced by the primary agent each round, overwrites the previous.
  • issues: reviewer returns a resolved list — those IDs are removed; reviewer returns a fresh issues list — runtime auto-assigns new IDs (I1, I2, …) and appends.
  • iteration: increments by 1 each round.

Context injection

IterationState is not an object the agent "sees". It is stitched into the user_text prompt by _build_primary_prompt() / _build_reviewer_prompt(). Every round, the agent receives a constant-sized context: original input + latest draft + unresolved issues. Context doesn't grow with iteration count.

03Loop mechanics

Two LLM calls per round. Reviewer emits structured JSON. Runtime decides when to stop.

for round in range(max_delegations):
    # 1. Primary agent drafts / revises
    draft = _stream_llm(primary_agent, input + state)
    state.draft = draft
    save_round_md(primary_dir, draft, round)
    send()                                       # done signal

    # 2. Reviewer critiques
    review = _stream_llm(reviewer_agent, input + state)
    feedback = parse_json_fenced(review)         # {issues, resolved}
    save_round_md(reviewer_dir, review, round)
    save_round_json(reviewer_dir, feedback, round)
    send()

    state.update(draft, feedback)                # drop resolved · assign IDs to new issues
    if not state.issues: break                   # empty list = pass

# Fell through max_delegations with unresolved issues?
# Log a warning, continue the pipeline with the last draft.
Copied ✓

The reviewer emits structured feedback via the _REVIEWER_OUTPUT_FORMAT JSON schema. The runtime — not the reviewer — checks whether issues is empty to decide pass/fail. The runtime mechanically maintains state and auto-assigns issue IDs (I1, I2, …); state management involves no LLM. When max_delegations is hit, the runtime logs a warning and proceeds with the last draft.

05A typical lifecycle

Three rounds, from fresh idea to "issues: []". This is the canonical shape of a Refine or Write session.

01

First draft, first critique

Explorer(idea) → draft v1
Critic(idea + v1) → {issues:[A,B,C]}
# runtime assigns IDs: I1=A, I2=B, I3=C
state = {draft: v1, issues: [I1,I2,I3], iteration: 1, _next_id: 4}
02

Revise; some resolved, one new

Explorer(idea + v1 + [I1,I2,I3]) → draft v2
Critic(idea + v2 + [I1,I2,I3]) → {issues:[D], resolved:[I1,I2]}
# runtime: I4=D · remove I1, I2
state = {draft: v2, issues: [I3,I4], iteration: 2, _next_id: 5}
03

Convergence — empty issues, break

Explorer(idea + v2 + [I3,I4]) → draft v3
Critic(idea + v3 + [I3,I4]) → {issues:[], resolved:[I3,I4]}
# issues is empty → break → save refined_idea.md / paper.md

06Reviewer JSON format

The reviewer's sole job is to emit this JSON. Runtime does the rest.

{
  "issues": [
    {
      "section": "Methodology",
      "problem": "DAG extraction feasibility unclear",
      "suggestion": "Add human-in-the-loop validation step"
    }
  ],
  "resolved": ["I1", "I3"]
}
Copied ✓
  • issues: new problems found this round. No id or severity fields — the runtime assigns IDs.
  • resolved: runtime-assigned IDs (like I1, I3) from the "Previously Identified Issues" block that are now fixed.
  • Pass decision: runtime checks if issues is empty after the update. No pass field needed.
  • format_issues() prefixes each with **I{n}** so the reviewer can reference them precisely.

07vs Research

Same philosophy, different execution shape.

Research Refine / Write
Loop strategy → decompose → execute → evaluate primary → reviewer → primary → reviewer
State task_results + plan_tree + score IterationState (draft + issues)
Driver ResearchStage._run_loop TeamStage._execute
Agent role One agent per task (many) Two fixed roles alternating
Communication via artifacts / DB via IterationState injected into prompt
Termination Evaluate emits no strategy_update issues empty, or max_delegations (warn + continue)

The shared core: Python owns the flow, the agent only executes one step, state lives in the runtime.

08Polish + Metadata sub-steps

Polish and Metadata are sub-steps of Write, not independent stages. WriteStage overrides TeamStage._execute(): after the Writer/Reviewer loop converges and paper.md is on disk, these two sub-steps run, and only then does the Write stage emit done.

Sub-step order (inside WriteStage._execute)

  1. Writer/Reviewer loop — standard TeamStage iteration via super(), produces paper.md.
  2. phase='polish' — single LLM call with POLISH_SYSTEM prompt. Uses polish_model (MAARS_POLISH_MODEL → write_model → google_model). Reads paper.md, returns polished prose.
  3. phase='metadata' — deterministic (no LLM). build_metadata_appendix() appends the meta block (generation date, models, config, …).
  4. Save paper_polished.md, emit Write stage done.

Compared to the loop

Writer/Reviewer loop Polish sub-step Metadata sub-step
Trigger TeamStage main loop After loop, phase='polish' After Polish, phase='metadata'
LLM iterative, up to max_delegations single call none (deterministic)
Input task data via DB tools paper.md polished prose + session meta
Output paper.md polished prose (in memory) append metadata block

Polish handles formatting cleanup, consistency, and metadata injection — work that doesn't benefit from iterative review. Because Polish is embedded in Write, the frontend progress bar has no separate "polish" node; the visible chain stays: refine / calibrate / strategy / decompose / execute / evaluate / write.
